# GPU 故障排查实战：三个生产环境剧本完整复盘

> **环境**：DGX A100 服务器，8x NVIDIA A100-SXM4-80GB，驱动 550.54.15，CUDA 12.4
> **运维守则**：先取证、后操作、先隔离、后维修

------

## 📋 剧本 A：Xid 48 显存物理损坏（经典硬件死刑）

### 1. 故障触发场景

- **告警信息**：监控系统弹窗 “GPU 3 Temp=N/A, Pwr=N/A, SM=ERR!”
- **业务影响**：跑在 GPU 3 上的训练任务崩溃退出
- **故障根因**：HBM2e 显存颗粒发生不可逆物理损坏，触发双比特 ECC 错误（Xid 48）

------

### 2. 第一步：`nvidia-smi` —— 确认异常卡

**命令**：

bash

```
nvidia-smi
```



**关键输出片段**：

text

```
|   3  NVIDIA A100-SXM4-80GB          On  | 00000000:1F:00.0 Off |                  N/A |
| ERR!  55C    P0             N/A  / 400W |      N/A  / 81920MiB |     N/A      Default |
```



**观察要点**：

- 状态栏显示 **`ERR!`**，而非正常空值
- 功耗显示 `N/A / 400W`，温度显示 `55C` 但不可信
- `Volatile Uncorr. ECC` 列为 `N/A`，无法读取
- 进程表中 GPU 3 对应条目为 `Not available due to GPU error`

**结论**：GPU 3 处于不可恢复错误状态，需进一步取证。

------

### 3. 第二步：`dmesg` —— 提取内核死亡日志

**命令**：

bash

```
dmesg -T | grep -i "1F:00\|Xid"
```



**关键输出片段**：

text

```
[Mon Apr 13 22:14:08 2026] NVRM: GPU 0000:1f:00.0: RmInitAdapter succeeded!
[Wed Apr 14 04:12:35 2026] NVRM: Xid (PCI:0000:1f:00.0): 48, pid=112348, Uncorrectable ECC error encountered on GPU. (Double Bit ECC)
[Wed Apr 14 04:12:35 2026] NVRM: Xid (PCI:0000:1f:00.0): 48, detail=DRAM single-bit ECC error threshold exceeded, page retirement failed.
[Wed Apr 14 04:12:36 2026] nvidia 0000:1f:00.0: AER: Uncorrected (Non-Fatal) error received
[Wed Apr 14 04:15:02 2026] NVRM: Xid (PCI:0000:1f:00.0): 79, GPU has fallen off the bus.
```



**日志解读**：

| 时间       | 事件                      | 含义                                  |
| :--------- | :------------------------ | :------------------------------------ |
| `22:14:08` | `RmInitAdapter succeeded` | 系统启动时 GPU 3 初始化成功，当时健康 |
| `04:12:35` | **Xid 48**                | 双比特 ECC 错误爆发，显存硬件损坏     |
| `04:12:35` | `page retirement failed`  | 坏块退役失败，损坏面积超出冗余容量    |
| `04:12:36` | `AER: Uncorrected`        | PCIe 层也检测到不可纠正错误           |
| `04:15:02` | **Xid 79**                | 在初始错误 2.5 分钟后，GPU 彻底失联   |

**结论**：死因明确为 Xid 48，属不可逆硬件故障。

------

### 4. 第三步：`nvidia-smi -q` —— 查看退役页面（伤残鉴定）

**命令**：

bash

```
nvidia-smi -q -i 3 | grep -A 10 "Retired Pages"
```



**关键输出片段**：

text

```
    Retired Pages
        Single Bit ECC             : 247
        Double Bit ECC             : 18
        Pending Page Blacklist     : Yes
        Remapped Rows              : 3
        Row Remap Failed           : Yes
```



**字段解读**：

| 字段                     | 值      | 健康阈值 | 含义                                           |
| :----------------------- | :------ | :------- | :--------------------------------------------- |
| `Single Bit ECC`         | 247     | < 10     | 因单比特错误过多而退役的页面数，极高值         |
| `Double Bit ECC`         | **18**  | **0**    | 因双比特错误退役的页面数，任何非零都是硬件实锤 |
| `Pending Page Blacklist` | Yes     | No       | 仍有待处理坏块，故障在蔓延                     |
| `Row Remap Failed`       | **Yes** | No       | 行重映射失败，死刑判决书上的红字               |

**结论**：显存已千疮百孔，不可修复，必须 RMA 更换。

------

### 5. 第四步：`nvidia-smi drain` —— 隔离故障卡

**命令**：

bash

```
nvidia-smi drain -p 1 -i 3
```



**输出**：

text

```
Successfully set GPU 00000000:1F:00.0 to drain state.
PCIe link state transition to L2 initiated.
GPU 3 is now marked as drained and will not be scheduled for new workloads.
```



**参数说明**：

| 参数    | 作用                                   |
| :------ | :------------------------------------- |
| `drain` | 将 GPU 标记为排水/禁用状态             |
| `-p 1`  | 同时将 PCIe 链路强制置为 L2 低功耗状态 |
| `-i 3`  | 指定 GPU 索引号为 3                    |

**验证**：再次执行 `nvidia-smi`，GPU 3 状态变为 `Drain`。

**目的**：从物理层切断故障卡的 PCIe 错误信号，防止干扰同机其他健康卡。

------

### 6. 第五步：业务恢复

- **8 卡分布式训练**：修改启动脚本 `CUDA_VISIBLE_DEVICES=0,1,2,4,5,6,7`，用剩余 7 卡重启训练
- **单卡任务**：调度器自动将 GPU 3 任务重调度至空闲卡

------

### 7. 第六步：RMA 返修

**提交材料**：

1. `nvidia-smi -q -i 3 > gpu3_fault.log`
2. `dmesg -T | grep -i "1F:00\|Xid" > gpu3_dmesg.log`

**厂商响应**：看到 Xid 48 + Double Bit ECC + Row Remap Failed，直接批准换卡。

**更换后验收**：

bash

```
nvidia-smi drain -p 0 -i 3   # 解除排水
dcgmi diag -r 3 -i 3         # 跑 3 级诊断
```

## 📋 剧本 B：SW Power Cap 误操作（人为配置错误）

### 1. 故障触发场景

- **用户反馈**：“GPU 2 训练速度比其他卡慢 40%”
- **初步观察**：`nvidia-smi` 无 `ERR!`，状态正常
- **故障根因**：上一班同事测试功耗限制脚本后忘记恢复，将 GPU 2 功耗墙锁死在 150W（A100 默认 400W）

------

### 2. 第一步：`nvidia-smi` —— 发现功耗异常

**命令**：

bash

```
nvidia-smi
```



**关键输出片段**：

text

```
|   2  NVIDIA A100-SXM4-80GB          On  | 00000000:17:00.0 Off |                    0 |
| N/A   43C    P0              149W / 150W |   68900MiB / 81920MiB |     58%      Default |
```



**对比其他健康卡**：

text

```
|   0  NVIDIA A100-SXM4-80GB          On  | 00000000:07:00.0 Off |                    0 |
| N/A   47C    P0              289W / 400W |   72450MiB / 81920MiB |     96%      Default |
```



**异常点识别**：

| 对比项     | GPU 2（异常）   | GPU 0（正常） |
| :--------- | :-------------- | :------------ |
| 功耗读数   | **149W / 150W** | 289W / 400W   |
| GPU 利用率 | **58%**         | 96%           |
| 性能状态   | P0              | P0            |

**初步判断**：GPU 2 撞到了 150W 的功耗墙，导致核心降频，利用率上不去。

------

### 3. 第二步：`nvidia-smi -q -d PERFORMANCE` —— 确认限制原因

**命令**：

bash

```
nvidia-smi -q -i 2 -d PERFORMANCE
```



**关键输出片段**：

text

```
    Performance State                     : P0
    Clocks Throttle Reasons
        SW Power Cap                      : Active
        HW Thermal Slowdown               : Not Active
        HW Power Brake Slowdown           : Not Active
    Power Readings
        Power Limit                       : 150.00 W
        Default Power Limit               : 400.00 W
        Enforced Power Limit              : 150.00 W
    Clocks
        Graphics                          : 765 MHz
        SM                                : 765 MHz
    Max Clocks
        Graphics                          : 1410 MHz
        SM                                : 1410 MHz
```



**关键字段解读**：

| 字段                      | 当前值      | 正常值/期望 | 解读                                |
| :------------------------ | :---------- | :---------- | :---------------------------------- |
| `SW Power Cap`            | **Active**  | Not Active  | 软件功耗限制已激活——人为操作实锤    |
| `Power Limit`             | 150.00 W    | 400.00 W    | 当前生效的功耗墙                    |
| `Default Power Limit`     | 400.00 W    | —           | 硬件默认上限，证明显卡本身支持 400W |
| `HW Thermal Slowdown`     | Not Active  | —           | 排除温度过高导致的降频              |
| `HW Power Brake Slowdown` | Not Active  | —           | 排除供电不足导致的硬件降频          |
| `Graphics Clock`          | **765 MHz** | ~1410 MHz   | 核心频率被压低了约 45%              |

**结论**：不是硬件故障，是人为配置错误。有人执行过 `nvidia-smi -pl 150 -i 2` 且忘记恢复。

------

### 4. 第三步：`nvidia-smi -pl` —— 恢复功耗上限

**命令**：

bash

```
nvidia-smi -i 2 -pl 400
```



**输出**：

text

```
Power limit for GPU 00000000:17:00.0 was set to 400.00 W from 150.00 W.
All done.
```



**参数说明**：

| 参数      | 作用                  |
| :-------- | :-------------------- |
| `-i 2`    | 指定 GPU 2            |
| `-pl 400` | 将功耗上限设置为 400W |

------

### 5. 第四步：验证恢复效果

**命令**：

bash

```
nvidia-smi -i 2
```



**恢复后输出片段**：

text

```
|   2  NVIDIA A100-SXM4-80GB          On  | 00000000:17:00.0 Off |                    0 |
| N/A   46C    P0              312W / 400W |   68900MiB / 81920MiB |     98%      Default |
```



**对比恢复前后**：

| 指标   | 恢复前      | 恢复后      |
| :----- | :---------- | :---------- |
| 功耗   | 149W / 150W | 312W / 400W |
| 利用率 | 58%         | 98%         |

**结论**：问题解决，GPU 2 恢复正常性能。

## 📋 剧本 C：Xid 79 + Riser 卡接触不良（连接件故障）

### 1. 故障触发场景

- **告警信息**：监控系统间歇性报 “GPU 6 Lost”，`nvidia-smi` 有时能看到 GPU 6，有时整卡消失
- **业务影响**：跑在 GPU 6 上的训练任务崩溃，调度器无法稳定使用该卡
- **故障根因**：Riser 卡金手指氧化 + 服务器风扇微振，导致 PCIe 接触不良，信号衰减触发链路降速并最终掉卡

------

### 2. 第一步：`nvidia-smi` —— 查看当前状态

**命令**：

bash

```
nvidia-smi
```



**关键输出片段**：

text

```
|   6  NVIDIA A100-SXM4-80GB          On  | 00000000:37:00.0 Off |                  N/A |
| Off  35C    P8             N/A  / 400W |      N/A  / 81920MiB |     N/A      Default |
```



**观察要点**：

- 状态栏显示 **`Off`**（而非 `P0`/`P8`），代表驱动已将其标记为离线
- 功耗、温度、显存全部 `N/A`，传感器通信中断
- 进程表中 GPU 6 对应条目为 `Not available - GPU offline`

------

### 3. 第二步：`lspci` —— 确认硬件层是否可见

**命令**：

bash

```
lspci | grep NVIDIA
```



**输出**（全部 8 卡均在）：

text

```
07:00.0 3D controller: NVIDIA Corporation GA100 [A100 SXM4 80GB] (rev a1)
0f:00.0 3D controller: NVIDIA Corporation GA100 [A100 SXM4 80GB] (rev a1)
...
37:00.0 3D controller: NVIDIA Corporation GA100 [A100 SXM4 80GB] (rev a1)
3f:00.0 3D controller: NVIDIA Corporation GA100 [A100 SXM4 80GB] (rev a1)
```



**关键发现**：

- GPU 6 的 Bus ID `37:00.0` **仍在 PCIe 总线上可见**
- 软件层（`nvidia-smi`）已放弃通信，但硬件层还能被枚举到
- **这完美符合 Xid 79 的特征**：GPU 没有彻底断电消失，而是 PCIe 通信间歇性故障

------

### 4. 第三步：`dmesg` —— 提取掉卡日志

**命令**：

bash

```
dmesg -T | grep -i "37:00\|Xid"
```



**关键输出片段**：

text

```
[Wed Apr 14 08:22:10 2026] NVRM: GPU at PCI:0000:37:00.0: GPU-e8f2a3b4-c5d6-4e7f-8a9b-0c1d2e3f4a5b
[Wed Apr 14 09:47:33 2026] nvidia 0000:37:00.0: AER: Corrected error received: 0000:37:00.0
[Wed Apr 14 09:47:33 2026] nvidia 0000:37:00.0: AER: PCIe Bus Error: severity=Corrected, type=Physical Layer, (Receiver ID)
[Wed Apr 14 09:47:35 2026] NVRM: Xid (PCI:0000:37:00.0): 79, pid=0, GPU has fallen off the bus.
[Wed Apr 14 09:47:35 2026] NVRM: GPU 0000:37:00.0: has fallen off the bus.
```



**日志解读**：

| 时间       | 事件                                   | 含义                                        |
| :--------- | :------------------------------------- | :------------------------------------------ |
| `09:47:33` | `AER: Corrected error, Physical Layer` | PCIe 物理层出现可纠正错误——信号质量开始恶化 |
| `09:47:35` | **Xid 79**                             | 2 秒后，驱动判定通信超时，GPU 掉卡          |

**结论**：掉卡由 PCIe 链路质量问题引发，非 GPU 本身损坏。

------

### 5. 第四步：`lspci -vvv` —— 查看链路协商状态

**命令**：

bash

```
sudo lspci -vvv -s 37:00.0 | grep LnkSta
```



**输出**：

text

```
                LnkSta: Speed 2.5GT/s (downgraded), Width x16 (ok)
```



**字段解读**：

| 字段         | 当前值      | 正常值            | 含义                 |
| :----------- | :---------- | :---------------- | :------------------- |
| `Speed`      | **2.5GT/s** | 16GT/s (PCIe 4.0) | 降级到 PCIe 1.0 速率 |
| `Width`      | x16 (ok)    | x16               | 通道数正常           |
| `downgraded` | **存在**    | 无                | 明确指示非正常降级   |

**根因分析**：

- Riser 卡金手指氧化/污损 → 高频信号衰减严重 → PCIe 控制器自动降速以求稳定
- 2.5GT/s 低速下仍无法维持连接 → 最终触发 Xid 79 掉卡

------

### 6. 第五步：`nvidia-smi drain` —— 隔离故障卡

**命令**：

bash

```
nvidia-smi drain -p 1 -i 6
```



**输出**：

text

```
Successfully set GPU 00000000:37:00.0 to drain state.
PCIe link state transition to L2 initiated.
GPU 6 is now marked as drained.
```



**验证**：`nvidia-smi` 中 GPU 6 状态变为 `Drain`。

------

### 7. 第六步：物理维护（需停机窗口）

**操作步骤**：

1. 服务器下电
2. 拔出 GPU 6 所在的 Riser 卡
3. 用橡皮擦轻轻擦拭金手指（去除氧化层）
4. 用精密电子清洁剂喷涂 PCIe 插槽
5. 重新插紧，确保卡扣完全锁死
6. 服务器上电

------

### 8. 第七步：恢复与验收

**解除隔离**：

bash

```
nvidia-smi drain -p 0 -i 6
```



**输出**：

text

```
Successfully cleared drain state for GPU 00000000:37:00.0.
PCIe link state restored to L0 (active).
```



**验证链路恢复**：

bash

```
sudo lspci -vvv -s 37:00.0 | grep LnkSta
```



**输出**：

text

```
                LnkSta: Speed 16GT/s (ok), Width x16 (ok)
```



**验证 GPU 上线**：

bash

```
nvidia-smi -i 6
```



**输出片段**：

text

```
|   6  NVIDIA A100-SXM4-80GB          On  | 00000000:37:00.0 Off |                    0 |
| N/A   35C    P8               36W / 400W |       0MiB / 81920MiB |      0%      Default |
```



**验收结论**：PCIe 链路恢复 Gen4 全速，GPU 6 重新加入资源池，故障解决。

------

### 9. 剧本 C 核心教训

| 要点                        | 说明                                              |
| :-------------------------- | :------------------------------------------------ |
| **Xid 79 ≠ GPU 损坏**       | 多数情况下是 PCIe 连接问题，先查 Riser 卡和线缆   |
| **`downgraded` 是关键线索** | 看到链路降速，直接指向物理层信号质量问题          |
| **隔离是必做动作**          | 即使计划物理维修，也先执行 `drain` 防止干扰其他卡 |

------

## 📋 三个剧本对照总结

| 对比维度              | 剧本 A（Xid 48）        | 剧本 B（功耗墙）    | 剧本 C（Xid 79 + Riser）   |
| :-------------------- | :---------------------- | :------------------ | :------------------------- |
| **故障类型**          | 硬件不可逆损坏          | 人为配置错误        | 连接件物理接触不良         |
| **`nvidia-smi` 状态** | `ERR!`                  | 正常 `P0`           | `Off`                      |
| **根因定位命令**      | `dmesg | grep Xid`      | `-q -d PERFORMANCE` | `lspci -vvv | grep LnkSta` |
| **是否需 RMA**        | 是                      | 否                  | 否（清洁即可）             |
| **隔离命令**          | `drain -p 1`            | 不适用              | `drain -p 1`               |
| **恢复命令**          | 更换硬件后 `drain -p 0` | `-pl 400`           | 物理清洁后 `drain -p 0`    |

------

> **运维心法总结**：
>
> 1. **先取证，后操作** —— 永远在执行 `drain` 或 `reset` 前把 `-q` 和 `dmesg` 保存下来
> 2. **先隔离，后维修** —— 坏卡的第一处理动作是切断其对总线的干扰
> 3. **软件现象，硬件根因** —— 看到 `ERR!` 想 Xid，看到 `Off` 查 PCIe，看到低利用率查功耗墙
> 4. **三个命令的调用顺序**：`nvidia-smi`（发现异常）→ `dmesg`（提取死因）→ `lspci`（确认物理层）→ `drain`（隔离）→ 针对性处置

这就是今天三个剧本的完整复盘。每一个步骤、每一条命令、每一段输出，都是生产环境中 GPU 运维的**肌肉记忆**。
